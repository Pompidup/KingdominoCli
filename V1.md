### Étape 2 — Domain : types métier & ports
- [x] Définir les types métier propres au CLI (pas de dépendance directe à l'engine) :
  - `AppState` : état global de l'application (screen courante, game state, cursor, erreurs, etc.)
  - `ScreenName` : `"title" | "config" | "game" | "results"`
  - `PlayerConfig` : configuration d'un joueur (nom, type humain/bot, niveau bot)
  - `GameConfig` : configuration d'une partie (mode, joueurs, extra rules)
  - `CursorState` : position, rotation, validité du placement
  - `TransitionState` : état de la transition entre joueurs
  - `ErrorState` : erreur contextuelle avec code et message
- [x] Définir les **ports** (interfaces) :
  - `GamePort` : interface d'accès au moteur de jeu (getModes, getExtraRules, createGame, addPlayers, startGame, chooseDomino, placeDomino, discardDomino, getResults, getValidPlacements, canPlaceDomino, calculateScore)
  - `BotPort` : interface pour les bots (playBotTurn, isBotTurn, getStrategyNames)
  - `UIPort` : interface pour le rendu (switchScreen, showError, showTransition, triggerAnimation)
  - `StatePort` : interface pour la gestion d'état (getState, dispatch, subscribe)
- 🧪 Tests unitaires : validation des types (compile-time), tests des type guards éventuels

## Notes techniques

- **Pattern Result** : les adapters retournent `{ ok: true, value: T } | { ok: false, error: { code, message } }` — jamais de throw vers la couche application
- **Immutabilité** : chaque mise à jour state produit un nouvel objet (l'engine est déjà stateless)
- **Séparation UI/logique** : les composants cligrid ne connaissent que les props qu'on leur passe, jamais l'engine directement
- **Events engine** : enregistrés une seule fois dans l'adapter, dispatch vers le state manager
- **Bot delays** : via `setTimeout` + Promise pour ne pas bloquer le rendering
